\begin{abstract}
This report presents a lightweight distributed file hosting system implemented in Python using FastAPI and asyncio. The system is composed of multiple peer nodes, each exposing a REST API for client operations and a TCP JSON-RPC channel for internal coordination. Files are replicated across nodes with configurable replication and quorum parameters. Each write is versioned with Lamport logical clocks, and conflicts are deterministically resolved using the tuple $(lamport\_counter, version\_node)$. A periodic health monitor detects peer failures and triggers recovery synchronization when nodes rejoin.

The implementation targets a practical balance between availability and consistency in partitioned environments. Write operations succeed when the configured write quorum is reached, while temporary divergence is repaired through background anti-entropy. The project includes containerized deployment through Docker Compose (three-node topology), a reproducible simulation script for failure and recovery scenarios, and a documented user workflow for upload, download, monitoring, and resilience validation.
\end{abstract}

\section{Concept}\label{concept}

The developed product is a distributed web service for file hosting. Each node provides a REST interface for client-facing upload, listing, metadata inspection, and download. Nodes also run an internal RPC service to replicate data, exchange metadata catalogs, and recover after failures.

Expected and tested interactions include:
\begin{itemize}
  \item frequent uploads of small and medium files,
  \item occasional downloads from any reachable node,
  \item operational checks of cluster health,
  \item fault-injection experiments in teaching or testing scenarios.
\end{itemize}

Distribution is needed for three main reasons:
\begin{itemize}
  \item \textbf{Fault tolerance}: files are replicated across peers, so node crashes do not imply immediate data loss.
  \item \textbf{Availability}: client requests can be served by multiple nodes; writes can still complete if quorum is met.
  \item \textbf{Resource sharing}: storage and traffic are spread across nodes, avoiding a single bottleneck.
\end{itemize}

\section{Requirements Elicitation and Analysis}\label{requirements}

\subsection*{Glossary}
\begin{itemize}
  \item \textbf{Logical path}: user-visible path used to identify a file in the system (for example, \texttt{docs/report.pdf}).
  \item \textbf{Replica}: copy of a file stored on a different node.
  \item \textbf{Write quorum}: minimum number of acknowledgements required for a successful write.
  \item \textbf{Catalog}: node metadata map from logical path to file metadata and version.
\end{itemize}

\subsection*{Functional requirements}
\begin{description}
  \item[FR1 -- File upload] A client shall upload a file with an optional logical path through a REST endpoint.
  \textbf{Acceptance criterion:} A \texttt{POST /files/upload} request returns HTTP 201 with metadata when quorum is reached.

  \item[FR2 -- File download] A client shall download the most recent file version from any node.
  \textbf{Acceptance criterion:} \texttt{GET /files/\{logical\_path\}/download} returns HTTP 200 and file bytes for existing files.

  \item[FR3 -- Metadata inspection] A client shall list node metadata and inspect per-file metadata.
  \textbf{Acceptance criterion:} \texttt{GET /files} and \texttt{GET /files/\{logical\_path\}/metadata} return JSON metadata when available.

  \item[FR4 -- Replication] On successful upload, the ingress node shall replicate to selected peers.
  \textbf{Acceptance criterion:} The upload response includes replica targets and acknowledged replicas.

  \item[FR5 -- Recovery synchronization] A node that becomes reachable again shall be synchronized automatically.
  \textbf{Acceptance criterion:} After restart of a failed node, it eventually serves the latest uploaded version.

  \item[FR6 -- Health visibility] Operators shall inspect node and peer health status.
  \textbf{Acceptance criterion:} \texttt{GET /health} reports node id, clock, quorum settings, and peer reachability.
\end{description}

\subsection*{Non-functional requirements}
\begin{description}
  \item[NFR1 -- Deterministic conflict handling] Concurrent versions shall be resolved consistently.
  \textbf{Acceptance criterion:} Version comparison based on $(lamport\_counter, version\_node)$ produces deterministic winners.

  \item[NFR2 -- Availability under partial failures] Write operations should continue while enough replicas are reachable.
  \textbf{Acceptance criterion:} With replication factor 3 and write quorum 2, writes succeed when at least one peer acknowledges.

  \item[NFR3 -- Bounded failure detection] Peer liveness checks shall run periodically.
  \textbf{Acceptance criterion:} Reachability state changes appear in health output after at most one monitoring period plus RPC timeout.

  \item[NFR4 -- Durability per node] Stored file bytes and metadata shall survive process restarts.
  \textbf{Acceptance criterion:} Restarting a container preserves files and metadata in mounted volumes.
\end{description}

\subsection*{Implementation constraints}
\begin{description}
  \item[IR1 -- Language/runtime] The system shall use Python 3.12 to align with course tooling and team familiarity.
  \textbf{Acceptance criterion:} The project runs in the provided Python 3.12 Docker image.

  \item[IR2 -- Deployment] The project shall be runnable with Docker Compose for reproducible evaluation.
  \textbf{Acceptance criterion:} A single \texttt{docker compose up -d --build} command starts the three-node cluster.

  \item[IR3 -- HTTP interface] Client-facing operations shall be exposed via REST/HTTP.
  \textbf{Acceptance criterion:} Upload, download, list, metadata, and health endpoints are callable through standard HTTP clients.
\end{description}

\subsection{Relevant Distributed System Features}
\label{ds-features}

\begin{itemize}
  \item \textbf{Transparency (partially relevant):} clients can use any node with the same REST API, but distribution details are intentionally visible in health and replication diagnostics.
  \item \textbf{Fault tolerance and dependability (highly relevant):} replication, periodic ping checks, and post-failure synchronization are central goals.
  \item \textbf{Scalability (moderately relevant):} architecture is peer-based and horizontally extensible, but no dynamic membership or sharding is implemented yet.
  \item \textbf{Security (currently limited):} no authentication/authorization layer is present; this is acceptable for a lab prototype but not production-ready.
  \item \textbf{Resource sharing (highly relevant):} storage is shared across nodes through replication and metadata coordination.
  \item \textbf{Openness/interoperability (relevant):} communication uses standard HTTP and JSON, easing integration with external clients.
  \item \textbf{Evolvability/maintainability (relevant):} components are modular (config, storage, RPC, cluster logic), enabling targeted improvements.
  \item \textbf{Performance and concurrency (relevant):} async I/O and parallel replication are used; the design targets simplicity over peak throughput.
  \item \textbf{Costs (relevant):} the stack is lightweight and containerized, minimizing infrastructure and onboarding overhead.
\end{itemize}

\section{Design}\label{design}

The design follows a symmetric peer-node architecture. Each node can receive client writes, store local data, replicate updates, and answer reads. This removes a single leader dependency and supports availability under partial failures.

\subsection{Architecture}\label{architecture}

The adopted style is \textbf{service-based peer-to-peer replication with dual interfaces}:
\begin{itemize}
  \item REST/HTTP for external client operations,
  \item TCP JSON-RPC for internal node-to-node coordination.
\end{itemize}

This split keeps client interactions simple while allowing richer internal operations (ping, catalog exchange, metadata fetch, content fetch, replication).

\subsection{Infrastructure}\label{infrastructure}

\begin{itemize}
  \item \textbf{Components:} 3 identical node services and any number of REST clients.
  \item \textbf{Per-node interfaces:} REST port 8000 (mapped to host 8001/8002/8003) and RPC port 9000 (mapped to 9001/9002/9003).
  \item \textbf{Storage:} each node has a dedicated persistent volume containing file blobs and \texttt{metadata.json}.
  \item \textbf{Placement:} all nodes run in a Docker Compose network for local deployment; the same design can be mapped to separate hosts.
  \item \textbf{Discovery:} static peer configuration is provided by the \texttt{PEERS} environment variable (JSON array with host and ports).
\end{itemize}

\subsection{Modelling}\label{modelling}

\begin{itemize}
  \item \textbf{Domain entities:} PeerConfig, FileVersion, FileMetadata, and logical file content.
  \item \textbf{State mapping:} each node stores local bytes under a deterministic file name and metadata indexed by logical path.
  \item \textbf{Domain events:} file uploaded, file replicated, peer failed, peer recovered, catalog synchronized.
  \item \textbf{Message types:} REST commands (upload/download/list) and RPC commands (ping, replicate\_file, get\_file\_metadata, get\_file\_content, get\_catalog).
  \item \textbf{System state:} file metadata catalog, local blob storage, Lamport clock value, and peer reachability map.
\end{itemize}

\subsection{Interaction}\label{interaction}

\begin{itemize}
  \item \textbf{Upload path:} client sends upload to one node; node writes locally and replicates in parallel to selected peers; response success depends on write quorum.
  \item \textbf{Read path:} node compares local metadata with peer metadata and fetches content from the newest source when needed.
  \item \textbf{Monitoring path:} background task periodically pings peers and updates reachability status.
  \item \textbf{Recovery path:} when a peer changes from unreachable to reachable, two-way synchronization is triggered.
  \item \textbf{Pattern summary:} request/response interactions plus periodic anti-entropy synchronization.
\end{itemize}

\subsection{Behaviour}\label{behaviour}

\begin{itemize}
  \item Nodes are stateful components: each one manages local storage, metadata, peer status, and a Lamport clock.
  \item The ingress node updates state on upload: it ticks the clock, creates a new version, and persists content.
  \item Replica nodes update state when \texttt{replicate\_file} RPC messages arrive, using Lamport update semantics.
  \item Storage accepts a version only if it is newer than the currently stored one.
  \item Recovery logic can both push local newer files and pull newer remote files.
\end{itemize}

\subsection{Data and Consistency Issues}\label{data-and-consistency-issues}

\begin{itemize}
  \item Persistent data includes file bytes and metadata fields (path, size, checksum, MIME type, version info).
  \item Storage model is file-system plus JSON metadata document (\texttt{metadata.json}) per node.
  \item Queries are local map lookups and catalog exchanges; there is no external DBMS.
  \item Concurrent operations are protected with thread locks in clock and storage classes.
  \item Shared cross-node data includes metadata versions and file content transported over RPC.
  \item Consistency model is eventual consistency with deterministic last-writer-wins ordering based on Lamport metadata.
\end{itemize}

\subsection{Fault-Tolerance}\label{fault-tolerance}

\begin{itemize}
  \item Replication is configurable by replication factor; replica targets are selected by hashing logical path and rotating peer list.
  \item Heart-beating is implemented via periodic \texttt{ping} RPC checks.
  \item Timeouts are enforced on every RPC operation.
  \item On write, if acknowledgements are below quorum, the node returns an error to the client.
  \item On peer recovery, background synchronization repairs stale or missing data.
  \item Errors in replication and recovery are isolated per peer and logged without crashing the node.
\end{itemize}

\subsection{Availability}\label{availability}

\begin{itemize}
  \item No dedicated cache layer is implemented.
  \item No explicit load balancer is included; clients may choose any node endpoint.
  \item Under network partitioning, the system remains writable when write quorum can be reached.
  \item Divergent replicas are reconciled when connectivity returns through catalog synchronization.
\end{itemize}

\subsection{Security}\label{security}

\begin{itemize}
  \item The current prototype does not implement authentication or authorization.
  \item No transport encryption (TLS) is configured by default.
  \item Integrity checks are partially supported through SHA-256 checksums stored in metadata.
  \item Security hardening is listed in \cref{future-works}.
\end{itemize}

\section{Implementation}\label{implementation}

\begin{itemize}
  \item \textbf{Network protocols:} HTTP for client API; raw TCP for node-to-node RPC.
  \item \textbf{Data representation:} JSON for REST and RPC payloads; file content encoded in Base64 for RPC transport.
  \item \textbf{Persistence/query style:} direct filesystem and JSON document handling; no SQL/NoSQL engine.
  \item \textbf{Authentication/authorization:} not yet implemented in this project iteration.
\end{itemize}

\subsection{Technological details}\label{technological-details}

\begin{itemize}
  \item Python 3.12 runtime.
  \item FastAPI as REST framework.
  \item Uvicorn as ASGI server.
  \item asyncio for concurrent replication, peer monitoring, and RPC operations.
  \item Docker and Docker Compose for reproducible three-node deployment.
  \item Main modules:
  \begin{itemize}
    \item \texttt{cluster.py}: orchestration, quorum logic, replication, and recovery,
    \item \texttt{storage.py}: persistent metadata and blob management,
    \item \texttt{rpc.py}: line-delimited JSON-RPC over TCP,
    \item \texttt{lamport.py}: logical clock semantics,
    \item \texttt{main.py}: REST endpoints and application lifecycle.
  \end{itemize}
\end{itemize}

\section{Validation}\label{validation}

\subsection{Automatic Testing}\label{automatic-testing}

\begin{itemize}
  \item Dedicated unit tests are not present yet.
  \item Integration and end-to-end behavior are automatically exercised by \texttt{scripts/simulate\_cluster.py}.
  \item The script tests this scenario:
  \begin{itemize}
    \item start cluster,
    \item upload first version,
    \item verify read from a replica,
    \item stop one node,
    \item upload second version during failure,
    \item restart failed node,
    \item verify that restarted node serves latest version.
  \end{itemize}
  \item Command to run automated scenario: \texttt{python3 scripts/simulate\_cluster.py}.
  \item Requirement coverage:
  \begin{itemize}
    \item FR1, FR2, FR4 via upload/read checks,
    \item FR5 via post-restart synchronization verification,
    \item NFR2 via quorum-based write with one node down.
  \end{itemize}
\end{itemize}

\subsection{Acceptance test}\label{acceptance-test}

\begin{itemize}
  \item Manual testing was performed through \texttt{curl} commands against all node endpoints.
  \item Tested behaviors:
  \begin{itemize}
    \item upload and metadata inspection,
    \item cross-node download,
    \item health endpoint changes after stopping and restarting containers,
    \item write failure behavior when quorum is not reachable.
  \end{itemize}
  \item Manual checks were useful to inspect response payloads and operational diagnostics interactively.
\end{itemize}

\section{Deployment}\label{deployment}

\begin{itemize}
  \item \textbf{Required software:} Docker Engine with Docker Compose support.
  \item \textbf{Startup command:}
\begin{verbatim}
docker compose up -d --build
\end{verbatim}
  \item \textbf{Expected outcome:} three containers running, exposed on REST ports 8001/8002/8003.
  \item \textbf{Configured environment:} each container sets node id, ports, data directory, quorum parameters, peer list, and RPC timeout.
  \item \textbf{Persistence:} named volumes \texttt{node1-data}, \texttt{node2-data}, and \texttt{node3-data} preserve data across restarts.
  \item \textbf{Standalone local run:} optionally possible without containers using virtual environment and Uvicorn, with distinct environment variables per node.
\end{itemize}

\section{User Guide}\label{user-guide}

\begin{itemize}
  \item Start the cluster:
\begin{verbatim}
docker compose up -d --build
\end{verbatim}

  \item Upload a file:
\begin{verbatim}
curl -X POST http://127.0.0.1:8001/files/upload \
  -F "file=@./sample.txt" \
  -F "logical_path=docs/sample.txt"
\end{verbatim}

  \item List metadata on a node:
\begin{verbatim}
curl http://127.0.0.1:8001/files
\end{verbatim}

  \item Download from another node:
\begin{verbatim}
curl http://127.0.0.1:8002/files/docs/sample.txt/download -o sample.txt
\end{verbatim}

  \item Check cluster health:
\begin{verbatim}
curl http://127.0.0.1:8001/health
\end{verbatim}

  \item Run full failure/recovery demo:
\begin{verbatim}
python3 scripts/simulate_cluster.py
\end{verbatim}
\end{itemize}

\section{Self-evaluation}\label{self-evaluation}

\textbf{Role and contribution.} I designed and implemented the full prototype, including API layer, replication logic, Lamport clock management, persistence layer, containerized deployment, and demonstration script.

\textbf{Strengths.}
\begin{itemize}
  \item Clear modular separation of concerns.
  \item Reproducible deployment with minimal setup.
  \item Practical handling of failures with automatic recovery.
  \item Deterministic version resolution in distributed updates.
\end{itemize}

\textbf{Weaknesses.}
\begin{itemize}
  \item Missing security layer (authentication, authorization, TLS).
  \item Limited automated testing depth (no unit test suite).
  \item Static peer configuration and absence of dynamic membership.
  \item No explicit load balancing or advanced placement policies.
\end{itemize}

\section{Future works}\label{future-works}

Planned improvements include:
\begin{itemize}
  \item introducing authentication and authorization (for example JWT plus role-based policies),
  \item enabling TLS for REST and RPC channels,
  \item supporting dynamic node membership and service discovery,
  \item introducing load balancing and request routing strategies.
\end{itemize}
